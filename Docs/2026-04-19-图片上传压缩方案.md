# 2026-04-19 图片上传自动压缩方案

## 一、背景

miniapp 端 `static/banners/banner-wallet.png` / `banner-gift.png` 单张接近 2 MB，**直接触发微信小程序主包 2 MB 上限**。进一步排查发现：

- 运营上传的 banner / 商品主图 / 优惠券图 / 券包图全部走 `POST /api/mediaassets/upload` 这一条链路
- 该接口 **完全不做压缩、不限大小、不转格式**（[MediaAssetsController.cs:154-187](backend/src/FaJuan.Api/Controllers/MediaAssetsController.cs#L154-L187)）
- 前端 [media-asset.ts:26-35](frontend/src/api/media-asset.ts#L26-L35) 原样 FormData 提交

⇒ 只要运营上传一张相机直出的 4-5 MB 图，后续引用它的所有页面（包括小程序）都会被拖累。所以**压缩必须在上传这一层拦截**，用户原话："从上传那里就要处理"。

## 二、现状盘点

| 环节 | 代码位置 | 现状 | 问题 |
|---|---|---|---|
| 前端上传 | [frontend/src/api/media-asset.ts:26](frontend/src/api/media-asset.ts#L26) | FormData 直传 | 无大小校验、无压缩 |
| 后端接收 | [backend/src/FaJuan.Api/Controllers/MediaAssetsController.cs:154](backend/src/FaJuan.Api/Controllers/MediaAssetsController.cs#L154) | 扩展名白名单 + 原样落盘 | 无大小限制、无内容校验、无压缩 |
| 存储位置 | `wwwroot/uploads/media/{yyyyMMddHHmmssfff}_{guid}{ext}` | 静态文件直出 | 文件大占磁盘+带宽 |
| 引用模型 | `MediaAsset.FileUrl` | banner / product / template / pack 都通过 `ImageAssetId` 关联 | 改压缩后新文件 URL 需保持契约 |

## 三、设计目标

| 指标 | 目标 |
|---|---|
| 运营上传原图大小上限 | ≤10 MB |
| 落盘后单张大小 | ≤100 KB |
| 落盘后长边像素 | ≤1080px（banner 场景 ≤1200px） |
| 统一目标格式 | WebP（质量 82），降级兜底 JPEG |
| 对现有 URL | 不改名、不改路径规则（只改文件内容+扩展名） |
| 对运营体验 | 无感知，后台选图上传后自动变小 |

## 四、三套方案对比

### 方案 A：纯后端压缩 ✅ 推荐起步

**流程**：前端原样传 → 后端读入 Stream → decode → resize → encode → 落盘 `.webp`。

| 优 | 缺 |
|---|---|
| 单点实现，规则强一致 | 后端单张 100-200ms CPU |
| 前端零改动 | 并发高时需加队列/限流 |
| 绕不过（curl 直传也会被压） | |

**技术选型**：
- `SixLabors.ImageSharp`（跨平台、无 GDI+ 依赖，**商业使用需购买 license**，2022 年起商业项目不能再用 Apache 2.0 版本）
- 或 `SkiaSharp`（Google 背书、licence 宽松 MIT，**推荐**）
- 或 `Magick.NET`（功能最全、包大）

**推荐 SkiaSharp**，免费、性能好、跨平台。

### 方案 B：纯前端压缩

**流程**：选图后 Canvas 缩放 → toBlob(webp) → FormData 上传。

| 优 | 缺 |
|---|---|
| 节省上传带宽 | 移动 Safari / 老浏览器 WebP 支持不一 |
| 后端零负担 | 可被绕过（curl 直传原图） |
| | 跨浏览器行为不稳 |

### 方案 C：混合 ✅ 生产最终形态

前端先压一次（降到 ~500 KB），后端再压一次（到 100 KB）。省流量 + 强保证，代价是两套代码。

## 五、推荐落地路径

### 阶段 1（本期，MVP）：只做方案 A

1. **加 NuGet 依赖**：`SkiaSharp` + `SkiaSharp.NativeAssets.Linux`（部署 Linux 时要加；Windows 默认 skia 已带）
2. **改 `MediaAssetsController.Upload`**：
   - 加 `[RequestSizeLimit(10_000_000)]`
   - 校验 `file.Length` ≤ 10MB（双保险，超过返 400）
   - 用 SkiaSharp 解码 → 按 `bucketType` 选长边上限（banner=1200，product=1080，shared=1080）→ 重编码 WebP quality=82
   - 落盘扩展名强制 `.webp`，`FileUrl` 返回 `.webp` 路径
3. **新增配置**（`appsettings.json`）：
   ```json
   "Uploads": {
     "MaxFileSizeBytes": 10485760,
     "DefaultLongSide": 1080,
     "BannerLongSide": 1200,
     "WebpQuality": 82
   }
   ```
4. **扩展名白名单**不变（`.jpg/.jpeg/.png/.gif/.webp` 都允许上传），但**出口统一 `.webp`**
5. **原有已上传图片不动**（老 URL 继续可访问）

### 阶段 2（可选）：加方案 B

前端在 `uploadMediaAssetFile` 前加一层：
- 图 > 1 MB 时走 Canvas 压缩到 1 MB 以内再提交
- 保留原始 `File` 对象作为 fallback（浏览器不支持 WebP 时直传）

### 阶段 3（可选）：存量回填

写一次性脚本把 `wwwroot/uploads/media/` 下现存大图重新压缩，更新 `MediaAsset.FileUrl`。**需停机或低峰期**，影响小程序线上显示。

## 七、风险 & 注意

- **透明图损失**：WebP 支持透明，但质量 82 下半透明边缘可能出现噪点。Banner 场景基本无影响，如果 UI 图标类素材需要无损，可为 `bucketType=icon` 留纯无损通道。
- **SkiaSharp 在 Linux Docker 部署**：需装 `libfontconfig1` / `libfreetype6`。Dockerfile 里 `apt-get install -y libfontconfig1`。
- **CPU 占用**：高并发上传场景建议套个 `SemaphoreSlim(Environment.ProcessorCount)` 限流，避免 CPU 打满。
- **大图上传超时**：10 MB 图在慢网下上传 >30s，前端要加进度条（`onUploadProgress`）。

## 八、决策点（需你确认）

1. 库选型：**SkiaSharp（推荐）** / ImageSharp / Magick.NET
2. 目标格式：**WebP 统一**（小程序/iOS/Android 都支持） / 保留原格式只压缩 / PNG 保留、JPEG/PNG 转 WebP
3. 是否做阶段 3 存量回填：**暂不做**（只压新图，老图按原样保留）/ 一次性压缩
4. 单张大小上限：**100 KB** / 150 KB / 200 KB
5. 长边上限：**banner 1200px、其他 1080px** / 全部 1080px / 全部 1440px
