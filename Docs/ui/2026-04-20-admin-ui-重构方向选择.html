后台重构方向选择

这一步只决定整体信息架构和视觉语气，不涉及任何业务逻辑变化。当前目标是把后台从“展示型页面”拉回“高频运营后台”，但保留必要质感与重点提醒能力。

A. 运营指挥台

保留管理后台的效率感，但顶部会有更强的总览和优先事项区。适合老板、运营、客服一起看。

优点
  • 整体高级感最强
  • 首页和重点页面更有“管理驾驶舱”感觉
  • 适合展示核心指标和待处理事项
代价
  • 如果做过头，容易再次偏展示型
  • 需要刻意压缩装饰，防止影响录入效率

B. 业务台账

把后台重心拉回“查、筛、录、改、核、跟”，整体更克制、更耐用，适合日常高频操作。

优点
  • 最像成熟 ERP / 运营后台
  • 对表格、筛选、弹窗编辑最友好
  • 最符合“只改 UI”边界
代价
  • 视觉惊艳度不如 A
  • 首页需要靠细节做质感，不靠大 hero

C. 风险监控台

把异常、待核销、待处理订单、失效模板这些信息提到前面，更偏“预警中心”。

优点
  • 适合订单、核销、分享追踪这类场景
  • 待办与异常优先级更清楚
  • 首页管理意味强
代价
  • 如果全站都用，会显得过于“告警系统”
  • 基础资料页未必匹配这种情绪

当前建议：选 B. 业务台账 作为主基调，再吸收 A 的少量质感，以及 C 的异常提醒方式。

原因很直接：你的后台核心是券模板、券包、订单、核销、分享追踪，都是高频操作页。主骨架应该稳、密、清楚，而不是继续大面积 hero 化。
