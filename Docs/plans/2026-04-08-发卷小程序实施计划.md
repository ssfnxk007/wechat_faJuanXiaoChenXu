# 发卷小程序 Implementation Plan

> **For agentic workers:** 推荐使用子 Agent 分任务执行。任务使用 checkbox (`- [ ]`) 语法跟踪。

**Goal:** 搭建一套可上线的发卷小程序系统，完成免费领券、指定发券、券包购买、支付发券、二维码核销与后台管理。

**Architecture:** 采用 `backend + frontend + miniapp + database` 四层结构。后端统一承担业务规则、支付回调和 ERP 核销；小程序负责用户端交互；后台负责运营配置；数据库使用 SQL Server 2008 R2 并结合 Code First 与 SQL 脚本双轨管理。

**Tech Stack:** `.NET 8`、`ASP.NET Core Web API`、`EF Core`、`Vue 3`、`TypeScript`、`Vite`、`uni-app`、`SQL Server 2008 R2`

---

## 一、建议的 Agent 拆分

- Agent 1：`backend/` 基础工程、实体、迁移、核心 API
- Agent 2：`frontend/` 后台管理框架与业务页面
- Agent 3：`miniapp/` 小程序登录、领券、卡包、券包购买页面
- Agent 4：`database/` SQL 脚本、初始化数据、部署说明
- 主 Agent：总控设计、接口契约、联调、验收、文档

## 四、UI 参考采集建议

你提到有一个参考小程序，后续可用以下方式做 UI 还原分析：

- 真机截图关键页面
- 录屏完整交互链路
- 用微信开发者工具辅助分析可见结构
- 用浏览器/自动化工具整理页面层级、文案与状态

如果参考对象是第三方线上小程序，通常不存在“直接导出全部 UI 源码”的通用官方工具。更现实的方法是：

- 截图 + 录屏
- 页面清单梳理
- 组件拆解
- 关键页面 1:1 复刻

## 五、执行建议

建议先按以下顺序启动多 Agent：

1. 后端基础 + 数据库
2. 小程序登录 + 我的券 + 领券中心
3. 后台券模板 + 商品 + 门店
4. 支付 + 券包 + 核销

完成每个阶段后，由主 Agent 统一做契约审查与联调。
