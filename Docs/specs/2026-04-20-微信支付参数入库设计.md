## 1. 背景与目标

目前发卷小程序的微信支付参数（AppId / MerchantId / MerchantSerialNo / PrivateKeyPemPath / ApiV3Key / NotifyUrl / EnableMockFallback）全部写在 `backend/src/FaJuan.Api/appsettings.json` 的 `WeChatPay` 节，通过 `IOptions<WeChatPayOptions>` 注入使用。并且商户私钥是指向本地文件的路径，部署时需要运维在服务器上落一份 `.pem` 文件。

这一做法的痛点：
- 修改参数要登录服务器改配置 + 重启进程
- 多机部署时 `.pem` 文件需要单独同步
- 没有后台页面，运营/测试无法自助切换模拟支付 / 正式支付

**本次目标**：把全部支付参数搬到 SQL Server 数据库；在管理后台"小程序设置"页新增 `支付参数` tab，让有权限的管理员可直接在页面上配置。

## 2.1 非目标 (out of scope)

- 不引入通用 KV 配置表（YAGNI）

- 不做平台证书管理页（当前 `WeChatPayService` 已经按需自动下载并缓存平台证书，无需暴露）

- 不做字段级审计日志（仅记录 `UpdatedAt`，不记录 `UpdatedBy` / 修改历史）
- 不对存入 DB 的私钥/APIv3 Key 做数据库级列加密；依赖 DB 访问控制和管理端鉴权保护（与项目现有敏感配置处理一致）

### 4.2 表结构 `wechat_pay_setting`

| 列 | 类型 | 说明 |

| `app_id` | `NVARCHAR(64)` NOT NULL DEFAULT `''` | 小程序 AppId |
| `merchant_id` | `NVARCHAR(32)` NOT NULL DEFAULT `''` | 商户号 |
| `merchant_serial_no` | `NVARCHAR(128)` NOT NULL DEFAULT `''` | 商户 API 证书序列号 |
| `private_key_pem` | `NVARCHAR(MAX)` NOT NULL DEFAULT `''` | 商户私钥 PEM 文本 |
| `api_v3_key` | `NVARCHAR(128)` NOT NULL DEFAULT `''` | APIv3 密钥 |
| `notify_url` | `NVARCHAR(512)` NOT NULL DEFAULT `''` | 回调地址 |
| `enable_mock_fallback` | `BIT` NOT NULL DEFAULT `1` | 未配置时是否回落模拟支付 |
| `updated_at` | `DATETIME2` NOT NULL | 最后修改时间 |

### 4.3 迁移策略

在 `AppDbContext` 注册 `DbSet<WeChatPaySetting>`，新增 `OnModelCreating` 配置：
- `HasKey(x => x.Id)`、`Property(x => x.Id).ValueGeneratedNever()`
- 列类型 & 长度按上表
- `HasData(new WeChatPaySetting { Id = 1, EnableMockFallback = true, UpdatedAt = new DateTime(2026, 4, 20, 0, 0, 0, DateTimeKind.Utc) })` 种子一行（`UpdatedAt` 必须是编译期常量，不能用 `DateTime.Now`，否则每次 `dotnet ef migrations` 都会生成无意义 diff）
- 生成迁移：`dotnet ef migrations add AddWeChatPaySetting`
- 启动时由 `Database.Migrate()` 自动应用（遵循 CLAUDE.md §6 的默认约定）

## 5. 服务层

### 5.1 `WeChatPaySettingsProvider`

`backend/src/FaJuan.Api/Infrastructure/WeChatPay/WeChatPaySettingsProvider.cs`

作为 **Scoped** 服务（跟随请求生命周期，避免跨请求共享 `DbContext`），内部持有一个 **静态**、**带时间戳** 的内存缓存：

```csharp
public class WeChatPaySettingsProvider(AppDbContext dbContext)
{
    private static WeChatPaySettingsSnapshot? _cached;
    private static DateTime _cachedAt;
    private static readonly SemaphoreSlim _lock = new(1, 1);
    private static readonly TimeSpan Ttl = TimeSpan.FromSeconds(30);

    public async Task<WeChatPaySettingsSnapshot> GetAsync(CancellationToken ct = default);
    public async Task<WeChatPaySettingsSnapshot> SaveAsync(WeChatPaySettingsUpdate update, CancellationToken ct = default);
    public void InvalidateCache();  // SaveAsync 内部自动调
}
```

- `GetAsync` 先查缓存（<30s 直接返回），miss 时从 `dbContext.WeChatPaySettings.AsNoTracking().SingleAsync()` 读并回填
- `SaveAsync` 采用「留空保留原值」策略（见 §6.3 DTO 行为）
- `SaveAsync` 成功后 `InvalidateCache()`，下一次读立即拉新值
- 缓存是静态字段，DbContext 是 scoped，两者寿命错开不会有"旧 context 持有结果"的 bug

### 5.2 `WeChatPayService` 改造

当前构造：

```csharp
public class WeChatPayService(HttpClient httpClient, IOptions<WeChatPayOptions> options)
{
    private readonly WeChatPayOptions _options = options.Value;
    ...
}
```

改为：

```csharp
public class WeChatPayService(HttpClient httpClient, WeChatPaySettingsProvider settingsProvider)
{
    private async Task<WeChatPaySettingsSnapshot> GetOptionsAsync(CancellationToken ct)
        => await settingsProvider.GetAsync(ct);

    public async Task<bool> IsConfiguredAsync(CancellationToken ct = default) { ... }
    // 每个对外方法加 CancellationToken 并内部调 GetOptionsAsync 获得参数
}
```

原 `IsConfigured()` 是同步的，改为异步 `IsConfiguredAsync()`；调用方全部更新（`PaymentsController:62`、`MiniAppController:923`）。

> 若调用链完全同步不便异步化，可在 `WeChatPaySettingsProvider` 暴露一个内部同步 `GetCachedOrThrow()`（仅当缓存命中时返回、否则触发一次 `GetAwaiter().GetResult()`），但优先尝试异步改造，实在不行再说。

### 5.3 平台证书缓存

`WeChatPayService` 内部已有 `_platformCertificates` 字段按商户证书序列号缓存平台证书 RSA。切换数据源不影响这一行为；只需在 `SaveAsync` 后的下一次请求中，由新 snapshot 的 `MerchantSerialNo` 触发必要的重下载（当前实现本身就按 key 查找、miss 时下载，天然兼容）。

### 6.2 DTO

`backend/src/FaJuan.Api/Contracts/AdminWeChatPaySettingsDto.cs`：

```csharp
public class AdminWeChatPaySettingsDto
{
    public string AppId { get; set; } = "";
    public string MerchantId { get; set; } = "";
    public string MerchantSerialNo { get; set; } = "";
    public string PrivateKeyDisplay { get; set; } = "";   // 非敏感占位，例如 "已配置" / ""
    public string ApiV3KeyDisplay { get; set; } = "";     // "已配置" / ""
    public string NotifyUrl { get; set; } = "";
    public bool EnableMockFallback { get; set; }
    public bool IsConfigured { get; set; }                // 便于前端提示
    public DateTime? UpdatedAt { get; set; }
}

public class SaveAdminWeChatPaySettingsRequest
{
    public string? AppId { get; set; }
    public string? MerchantId { get; set; }
    public string? MerchantSerialNo { get; set; }
    public string? PrivateKeyPem { get; set; }  // null / "" = 不修改
    public string? ApiV3Key { get; set; }       // null / "" = 不修改
    public string? NotifyUrl { get; set; }
    public bool? EnableMockFallback { get; set; }
}
```

### 6.3 敏感字段行为

- `AppId / MerchantId / MerchantSerialNo / NotifyUrl / EnableMockFallback`：**明文双向**。GET 按实际值返回，PUT 直接覆盖（不区分 null 与空串：`null` 保持原值、空串表示清空——这里简化为 `null = 不改，非 null = 覆盖`）
- `PrivateKeyPem / ApiV3Key`：**只写不读**。GET 返回 `PrivateKeyDisplay = 原值非空 ? "已配置" : ""`；PUT 时 `null || trim=="" || == "已配置"` 三种情况都视为"不修改"，否则覆盖。注意：`"已配置"` 这个占位字符串不会被 §6.4 的 PEM 校验误当作有效 PEM 吸收，因为校验要求包含 `BEGIN PRIVATE KEY` / `END PRIVATE KEY` 标记
- `IsConfigured`：由后端判断所有必填字段（AppId / MerchantId / MerchantSerialNo / PrivateKeyPem / ApiV3Key / NotifyUrl）全非空才为 `true`

### 6.4 校验

- `MerchantId`：只允许数字 & 长度 <= 32
- `NotifyUrl`：必须以 `https://` 开头（或留空）
- `PrivateKeyPem`：如果非 null/非空，需包含 `BEGIN PRIVATE KEY` 与 `END PRIVATE KEY`（正则粗检即可，真正错误由微信接口调用时报错）
- 校验失败返回 `400` + `ApiResponse.Failure("...")`
