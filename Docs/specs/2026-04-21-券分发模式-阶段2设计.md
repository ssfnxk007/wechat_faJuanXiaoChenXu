# 券分发模式（Coupon Distribution Mode）· 阶段 2 设计

- 日期：2026-04-21
- 作者：Codex
- 状态：待实施
- 前置：`Docs/specs/2026-04-21-券分发模式-阶段1设计.md`

---

## 1. 目标

阶段 2 目标是把阶段 1 已完成的“券分发建模”推进到**小程序可售卖**：

- `PaidStandalone` 的**单张售卖券**可以在小程序展示、下单、支付、支付成功后发券
- `TemplateType = Product` 且可售卖的**商品券**可以在小程序展示、下单、支付、支付成功后发券
- 券包购买链路保持可用，不回归
- **ERP 对接 / ERP 核销**不在本阶段实现，但商品券要保留“待履约 / 待 ERP 处理”的业务语义

---

## 2. 本阶段范围

### 2.1 包含

1. 扩展 `CouponOrder`，同时支持“券包订单”和“单张券订单”
2. 小程序 `mall` 页增加：
   - 主推券包
   - 单张售卖券
   - 商品券专区
3. 单张券 / 商品券详情、下单、支付、支付成功发券
4. 订单列表 / 订单详情兼容显示两类订单来源
5. 商品券发券后记录“待履约”状态，供后续 ERP 阶段接入

### 2.2 不包含

- ERP 自动发货
- ERP 核销接口对接
- 线下履约状态回写
- 复杂售后 / 退款细分策略重构
- 新开独立商城页或新 tab

## 4. 页面与交互设计

## 4.1 `mall` 页结构

现有 `miniapp/coupon-miniapp/pages/mall/index.vue` 继续作为商城页，不新建平级页面。

页面区块调整为：

1. **主推券包**
   - 维持现有大卡片样式
   - 用于组合售卖，不改主视觉密度
2. **单张售卖券**
   - 展示 `DistributionMode = PaidStandalone`
   - 排除 `TemplateType = Product`
   - 卡片重点展示：券名、券类型、售价、优惠力度、适用说明
3. **商品券专区**
   - 展示 `DistributionMode = PaidStandalone && TemplateType = Product`
   - 卡片重点展示：券名、目标商品摘要、售价、履约提示
4. **精选商品**
   - 维持现有商品格子

## 4.2 排版原则

用户已明确接受“在 `mall` 页增加单张券区块”，但要求注意排版，因此本阶段采用以下约束：

- 券包区仍保持第一屏核心位置，不被新模块挤掉
- “单张售卖券”与“商品券专区”使用**列表卡片**，不与商品共用两列宫格，避免信息过密
- “精选商品”继续保持双列商品卡，因为它更适合图片主导展示
- 单张券卡片高度控制在与券包卡片相近的视觉节奏内，避免页面突然出现超高列表
- 商品券卡片必须有“待履约 / 待 ERP 处理”的轻提示，不让用户误以为是普通抵扣券

## 4.3 购买流转

### 单张售卖券

`mall` 页卡片 -> 券详情页 -> 点击购买 -> 创建订单 -> 拉起支付 -> 支付成功 -> 发券到“我的券”

### 商品券

`mall` 页商品券专区卡片 -> 券详情页 -> 点击购买 -> 创建订单 -> 拉起支付 -> 支付成功 -> 发券到“我的券” -> 券详情展示“待履约 / 待 ERP 处理”

## 5. 数据模型设计

## 5.1 `CouponOrder` 扩展

当前 `CouponOrder` 只支持券包订单：

```csharp
public long CouponPackId { get; set; }
```

阶段 2 改为：

```csharp
public long? CouponPackId { get; set; }
public long? CouponTemplateId { get; set; }
```

业务约束：

- `CouponPackId` 与 `CouponTemplateId` 必须二选一
- 券包订单：`CouponPackId != null && CouponTemplateId == null`
- 单张券订单：`CouponPackId == null && CouponTemplateId != null`

## 5.2 商品券履约状态

本阶段不改 ERP 流程，但商品券不能和普通券完全无差别。

推荐新增轻量字段到 `UserCoupon`：

```csharp
public CouponFulfillmentStatus FulfillmentStatus { get; set; } = CouponFulfillmentStatus.None;
```

枚举建议：

```csharp
public enum CouponFulfillmentStatus
{
    None = 0,              // 普通券，无履约语义
    PendingFulfillment = 1,// 商品券，待 ERP / 人工履约
    Fulfilled = 2,         // 后续 ERP 阶段使用
    Cancelled = 3          // 后续退款/关闭时可用
}
```

规则：

- 普通单张售卖券：`FulfillmentStatus = None`
- 商品券：支付发券时写入 `PendingFulfillment`

> 说明：如果不想本阶段改 `UserCoupon`，也可以加独立表 `ProductCouponFulfillment`。但当前项目代码体量下，直接在 `UserCoupon` 增轻量状态字段更实用，后续 ERP 对接再决定是否拆表。

## 6. 后端接口设计

## 6.1 `mall` 数据接口

当前小程序 `mall` 页通过两个接口拼数据：

- `/api/miniapp/coupon-packs`
- `/api/miniapp/products`

阶段 2 建议新增聚合接口：

### `GET /api/miniapp/mall`

返回：

```json
{
  "packs": [],
  "standaloneCoupons": [],
  "productCoupons": [],
  "products": []
}
```

这样小程序 `mall` 页一次请求就能拿到四个区块的数据，避免前端再拼 4 次调用。

筛选规则：

- `packs`：现有券包逻辑
- `standaloneCoupons`：`IsEnabled && DistributionMode == PaidStandalone && TemplateType != Product`
- `productCoupons`：`IsEnabled && DistributionMode == PaidStandalone && TemplateType == Product`
- `products`：现有商品逻辑

## 6.2 单张券详情

新增或复用小程序券模板详情接口，使未领取的可售卖券也可查看：

### `GET /api/miniapp/sale-coupons/{id}`

返回字段至少包括：

- `CouponTemplateId`
- `Name`
- `TemplateType`
- `DistributionMode`
- `SalePrice`
- `DiscountAmount`
- `ThresholdAmount`
- `Remark`
- `ImageUrl`
- `IsAllStores`
- `ProductSummary`
- `FulfillmentHint`（商品券时显示“支付成功后待 ERP 处理”）

约束：

- 仅允许 `DistributionMode == PaidStandalone`
- 非售卖券返回 404 或 400

## 6.3 创建订单

当前小程序下单接口：

### `POST /api/miniapp/orders`

当前只收：

```json
{ "couponPackId": 1 }
```

阶段 2 建议扩为统一订单创建接口：

```json
{
  "couponPackId": 1,
  "couponTemplateId": null
}
```

或

```json
{
  "couponPackId": null,
  "couponTemplateId": 12
}
```

校验规则：

- 两者必须二选一
- 券模板下单时要求：
  - `DistributionMode == PaidStandalone`
  - `IsEnabled == true`
  - `SalePrice > 0`
  - 满足每用户限购规则

## 6.4 支付

现有 `/api/miniapp/orders/{id}/pay` 继续复用，不新开第二套支付接口。

需要改造点：

- 订单是券包时，保留现逻辑
- 订单是单张券时，支付描述改为 `单张券订单-<OrderNo>`
- 支付成功后，`OrderPaymentService` 能分流处理两种订单来源

## 6.5 支付成功发券

`OrderPaymentService.MarkOrderPaidAsync` 现在只会按 `CouponPackId` 发券。

阶段 2 要改成：

- 如果订单来源是券包：按 `CouponPackItems` 批量发券
- 如果订单来源是单张券：按 `CouponTemplateId` 发 1 张券

商品券发券时：

- `UserCoupon.Status = Unused`
- `UserCoupon.FulfillmentStatus = PendingFulfillment`

普通单张券发券时：

- `UserCoupon.Status = Unused`
- `UserCoupon.FulfillmentStatus = None`

## 7.2 数据层

- mock fallback 同步扩展出：
  - `standaloneCoupons`
  - `productCoupons`

## 7.3 卡片组件

建议新增两个轻量组件，而不是把券包卡硬改成万能卡：

- `components/StandaloneCouponCard.vue`
- `components/ProductCouponCard.vue`

原因：

- 单张售卖券强调“售价 + 券权益”
- 商品券强调“售价 + 目标商品 + 待履约提示”
- 这两类信息结构都与券包卡不同，强行复用会让模板和样式变乱

## 7.4 详情页

建议新增单张券详情页：

- `pages/sale-coupon/detail`

它同时服务：

- 普通单张售卖券
- 商品券

页面差异通过 `TemplateType` 控制：

- 普通券：展示优惠、门店范围、有效期
- 商品券：额外展示目标商品摘要与履约说明

## 7.5 订单页

`pages/order/list` 与 `pages/order/detail/result` 需要兼容展示：

- 订单来源：券包 / 单张券
- 名称：`CouponPackName` 或 `CouponTemplateName`
- 商品券订单在结果页和订单详情页显示“待履约 / 待 ERP 处理”

## 9. 风险与控制

- **订单模型改造风险最高**：`CouponOrder` 从单一来源变成双来源，所有订单列表、详情、支付、退款都要检查空值分支
- **商品券语义风险**：如果不补“待履约”状态，用户会误以为商品券和普通抵扣券完全一样
- **mall 页排版风险**：新增两个区块后页面会变长，必须保持“券包优先、售卖券次之、商品次之”的信息层级

## 10. 本阶段交付结果

阶段 2 完成后，业务上会得到：

- 后台已配置为 `PaidStandalone` 的券，终于在小程序有正式售卖入口
- 商品券可以从“只建模”进入“可下单 / 可支付 / 可发券”
- 券包链路不回退
- ERP 还没接，但商品券已经具备后续接 ERP 的状态承载点
