# 发卷小程序设计说明

## 1. 项目目标

构建一套面向微信生态的发券系统，覆盖以下第一阶段能力：

- 微信小程序登录并自动建档用户
- 免费领券、指定用户发券、按门店/渠道/活动发券
- 券包售卖、微信支付、支付成功自动发券
- 小程序展示用户券二维码，ERP 扫码后调用接口核销
- 提供后台管理端完成券、商品、门店、券包、订单、核销与导入管理

第二阶段目标：接入企业微信做客户触达、活动通知与导流，但不作为第一阶段的核心账户体系。

## 2. 项目结构

项目从 0 新建，目录结构如下：

- `backend/`：`.NET 8 Web API`
- `frontend/`：`Vue 3 + TypeScript + Vite` 后台管理系统
- `miniapp/`：`uni-app` 微信小程序
- `database/`：`SQL Server 2008 R2` SQL 初始化与补充脚本
- `Docs/`：设计、计划、工作日志、接口文档

第一阶段不创建 `pos-client/`。

## 4. 核心业务模型

### 4.1 用户

用户进入小程序后通过微信登录自动建档。

第一阶段身份策略：

- 主身份：`MiniOpenId`
- 辅助匹配：`Mobile`
- 预留字段：`UnionId`、`OfficialOpenId`

说明：

- 当前 ERP 仅有公众号 `OpenId`，不能直接转换为小程序 `OpenId`
- 第一阶段指定发券采用“手机号辅助匹配”方案
- 历史导入名单中未完成匹配的用户进入“待匹配”状态，待其首次进入小程序并绑定手机号后自动补发

### 4.2 券模板

第一阶段支持 4 类券：

- 无门槛券
- 指定商品券
- 满减券
- 新人券

预留扩展类型：

- 折扣券
- 买赠券
- 兑换券

### 4.3 用户券实例

用户真正持有并可出示二维码核销的是“用户券实例”，每张券实例具备唯一券码。

### 4.4 券包商品

券包商品是可售卖的业务商品，例如：

- 5 元券包：2 张 5 元无门槛券 + 3 张指定商品券
- 10 元券包：4 张满减券 + 2 张指定商品券

支付成功后，系统根据券包明细批量生成用户券实例。

## 5. 业务规则

### 5.1 用户与登录

- 用户首次进入小程序后，使用微信登录自动建档
- 允许后续绑定手机号
- 手机号用于匹配后台导入的指定发券名单

### 5.2 发券方式

第一阶段支持以下发券方式：

- 公开领取
- 指定用户主动发放
- 按门店范围发放
- 按渠道范围发放
- 按活动范围发放
- 购买券包后自动发券

### 5.3 新人券规则

- 新人券为独立券种
- 每个 `MiniOpenId` 终身只能领取一次
- 与券是否过期、是否已核销无关，不会重置资格

### 5.4 有效期规则

所有券都必须配置有效期。

第一阶段支持：

- 固定日期范围
- 领取后 N 天有效
- 领取后立即生效

用户券实例必须记录：

- `领取时间`
- `生效时间`
- `失效时间`

### 5.5 商品券规则

- 后台可以自己维护商品资料
- 商品资料包含“ERP 商品唯一码”字段
- 指定商品券通过 ERP 商品唯一码绑定商品

### 5.6 门店可用范围

券支持配置：

- 全部门店可用
- 指定门店可用

ERP 核销时必须上传门店标识，后端按券规则校验。

### 5.7 指定发券导入规则

后台支持 Excel 导入指定发券名单，导入字段建议包括：

- 姓名
- 手机号
- 公众号 OpenId（可选）
- 门店
- 渠道
- 活动
- 券模板
- 数量

导入结果状态：

- 已匹配并已发券
- 待匹配
- 匹配失败
- 已失效

### 5.8 核销规则

- 小程序展示二维码
- 二维码内容仅包含随机唯一券码
- ERP 扫码后调用后端核销接口
- 后端校验：
  - 券是否存在
  - 是否已生效
  - 是否已过期
  - 是否已使用
  - 是否可在当前门店使用
- 校验成功后写入核销记录

### 5.9 券包售卖与退款规则

- 券包支持微信支付
- 支付成功后自动发券
- 发券必须具备幂等保护，防止微信重复回调导致重复发券

退款规则：

- 仅支持整单退款
- 仅当订单下所有券都未使用时才允许退款
- 若任意一张券已核销，则整单不可退款
- 退款成功后，回收该订单对应的全部未使用券

## 8. Code First 建表策略

本项目允许采用 `EF Core Code First` 生成模型，并由程序统一管理数据结构。

但考虑到数据库为 `SQL Server 2008 R2`，需注意：

- 实体设计尽量保守，避免依赖新版本 SQL Server 特性
- 生成迁移时控制字段类型，避免出现与 2008 R2 不兼容的 SQL 语法
- 重要初始化数据仍输出到 `database/` 中，便于部署与回放
- 菜单、权限、字典等基础数据建议保留可执行 SQL 脚本

建议实际落地方式：

- 用 `Code First` 维护实体与迁移
- 同时在 `database/` 输出首版建表 SQL 与初始化 SQL

## 9. 外部系统对接设计

### 9.1 ERP

ERP 负责扫码并调用后端核销接口。

建议 ERP 调用接口时传入：

- 券码
- 门店编码
- 操作人
- 设备号（可选）

后端返回：

- 是否核销成功
- 失败原因
- 券信息摘要
- 用户信息摘要

### 9.2 微信生态

第一阶段使用：

- 小程序登录
- 手机号能力
- 微信支付

第二阶段规划：

- 企业微信触达
- 活动导流
- 客户运营

企业微信不作为第一阶段核心发券账户体系。

## 10. UI 设计策略

### 10.1 小程序 UI

你会提供参考小程序，后续可基于参考 UI 做结构分析与视觉拆解。

建议采用方式：

- 真机截图
- 页面录屏
- 开发者工具辅助抓取页面结构
- 结合交互路径输出页面清单与组件清单

对于第三方线上小程序，通常没有官方“直接导出完整 UI 源码”的标准工具；更可行的是基于截图、录屏、交互分析和页面复刻。

### 10.2 后台 UI

后台优先采用成熟业务后台布局：

- 登录页
- 搜索卡片 + 表格卡片
- 弹窗式新增/编辑
- 主从表场景使用弹窗详情

### 第二阶段

## 12. 风险与约束

- `SQL Server 2008 R2` 较老，建模和迁移策略要保守
- 微信与企业微信能力需区分，不能把公众号 `OpenId` 直接转换为小程序 `OpenId`
- 指定发券若无小程序身份，需要依赖手机号后匹配补发
- 支付与发券必须重点处理幂等，避免重复回调带来重复发券

## 13. 结论

第一阶段采用“从 0 新建三端”的方式最合适：

- `backend/`
- `frontend/`
- `miniapp/`

并以“领券 + 卖券包 + ERP 核销”为核心闭环，后续再把企业微信作为增长与运营模块接入。
