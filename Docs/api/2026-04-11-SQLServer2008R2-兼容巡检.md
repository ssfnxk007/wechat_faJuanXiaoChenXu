## 本轮巡检结论

### 1. 高风险分页语法

- 已复扫业务代码中的数据库分页查询，未再发现残留的 LINQ `Skip(...).Take(...)` 用法。
- 当前唯一残留的 `Skip(1)` 位于文本导入逻辑，不参与数据库 SQL 生成，不属于兼容问题。

### 2. 模型字段类型

- 当前实体主要使用：`datetime`、`decimal(18,2)`、`nvarchar(n)`、`nvarchar(max)`、`bit`、`bigint`、`int`
- 以上类型对 SQL Server 2008 R2 兼容。

### 3. 迁移与 DDL 风险

- 现有迁移未发现以下高风险语法：
  - `DROP TABLE IF EXISTS`
  - `CREATE OR ALTER`
  - `FILTERED INDEX` / `HasFilter(...)`
  - `SEQUENCE`
  - `JSON` 相关 SQL
  - `datetimeoffset` / `datetime2` 精度依赖
  - `FromSqlRaw` / `ExecuteSqlRaw` 中的现代 SQL 片段

### 4. 已知兼容保障

- `Program.cs` 已设置：`UseCompatibilityLevel(100)`
- 分页扩展已用嵌套 `TOP` 方式模拟分页，避免 `OFFSET/FETCH`

## 当前仍需持续注意的规则

### 必须遵守

- 所有数据库分页查询禁止直接写 `Skip/Take`
- 所有新增迁移 SQL 交付前必须人工复核一次 SQL Server 2008 R2 兼容性
- 所有新控制器若有分页，一律复用 `ApplyLegacyPaging`

### 后续巡检建议

- 每次新增接口后，优先检查：
  - 是否引入新的 `Skip/Take`
  - 是否引入新的 `GroupBy + 分页 + Join` 组合查询
  - 是否引入新的高版本 SQL Server DDL 或函数
- 每次新增迁移后，导出 SQL 并对照 2008 R2 语法手审一次
